## 객체란
객체 지향적 설계를 이야기하려면 먼저 '객체'라는 것부터 이해하고 넘어가야 할 것 같았다. 제임스 고슬링이 작성한 자바 언어 설명서(The Java Language Specification)에서는 객체(object)를 클래스의 인스턴스나 배열이라고 정의한다. 이렇게 들어도 사실 잘 이해가 되지 않아서, 나만의 방식으로 객체를 다시 고민해보았다. 우리는 개발을 하면서 프로그래밍에서의 객체에 대해 자연스럽게 많이 생각하게 되는데, 그 과정에서 나는 객체를 기능이나 고유한 속성들의 단위라고 이해하게 되었다.

## 객체를 지향하라!
객체를 지향한다는 것은 곧 기능이나 고유한 속성들의 단위로 분리하라는 의미라고 생각한다. 그래서 Java에서 하나의 class가 너무 많은 역할을 맡게 되면 유지보수가 어려워지고, 다른 사람이 내 코드를 이해하기도 힘들어진다. 객체 지향은 결국 이런 어려움을 해소하기 위한 접근이라고 생각한다.

## 추상화?
추상화는 지겹도록 들어왔을 개념이다. Java에서의 추상화(Abstraction)는 공통된 속성이나 행위(기능)들을 모아 클래스로 정의하는 것을 말한다. 추상화는 복잡성을 줄여주고 필요한 정보만 노출시켜주기 때문에, 유지보수와 간결한 코드 작성에 있어 없어서는 안 되는 개념이다.

## 객체 지향적 설계
결론적으로 객체 지향적 설계를 하기 위해서는 S.O.L.I.D 원칙을 지켜야 한다. SOLID를 정리하면서 다시 한번 이 원칙들을 머릿속에 정리하는 시간을 가져보려고 한다.

### ✔️ SRP (Single Responsibility Principle) 단일 책임 원칙
### ✔️ OCP (Open Closed Principle) 개방 폐쇄 원칙
### ✔️ LSP (Liskov Substitution Principle) 리스코프 치환 원칙
### ✔️ ISP (Interface Segregation Principle) 인터페이스 분리 원칙
### ✔️ DIP (Dependency Inversion Principle) 의존 역전 원칙

## 📌 SRP 단일 책임 원칙
클래스의 책임과 역할은 하나로 한다. 즉, 클래스를 설계할 때 하나의 기능을 담당하여 하나의 책임을 수행하는 데에 집중해야 한다.

## 📌 OCP 개방 폐쇄 원칙
중요한 포인트는 '확장에는 열려있다'와 '변경에는 닫혀있다'는 것이다.

- ✔️ **확장에 열려있다**: 기능 추가에 있어 클래스를 간편하게 구현할 수 있어야 하며, 기존 코드 수정은 최소화할 수 있어야 한다는 의미다.
- ✔️ **변경에 닫혀있다**: 객체를 직접 수정하지 않고도 변경 사항을 적용할 수 있어야 한다는 의미다.

## 📌 LSP 리스코프 치환 원칙
개인적으로 다섯 원칙 중 가장 이해하기 어려웠던 원칙이다.

> "서브 타입은 언제나 자신의 기반 타입(base type)으로 교체할 수 있어야 한다." — 로버트 C. 마틴

## 📌 ISP 인터페이스 분리 원칙
특정 상황에 관련된 기능을 작동시키는 메서드만을 제공하는 것이다. SRP와의 차이는, SRP는 클래스의 단일 책임을 강조하는 반면 ISP는 인터페이스의 단일 책임을 강조한다는 점이다. 다만 인터페이스는 추상적인 존재이기 때문에 여러 역할을 갖는 데에 제약이 없다. 가장 중요한 것은 관련된 기능끼리 인터페이스에 모으되, 하나의 인터페이스가 너무 커지지 않도록 하는 것이다.

## 📌 DIP 의존 역전 원칙
고수준 모듈은 저수준 모듈에 의존해서는 안 된다. 둘 다 추상화(추상 클래스나 인터페이스)에 의존해야 한다. 즉, 예를 들어 서비스 클래스가 특정 DB 구현체를 직접 참조하는 게 아니라 Repository 인터페이스에 의존하게 만들어야 한다는 뜻이다. 이렇게 설계하면 나중에 DB 구현체가 바뀌더라도 서비스 클래스의 코드는 그대로 유지할 수 있다.
