# 시작은 후배의 제안에서

이 프로젝트에 합류하게 된 계기는 우연이었습니다. 동아리에서 알게 된 기계공학과와 컴퓨터SW학과를복수전공하고 있는 후배가 자기 졸업 프로젝트를 함께 해보지 않겠냐고 제안을 해왔고, 그 자리에서 아이디어를 처음 들었습니다. 방향은 어느 정도 잡혀 있는 상태인 줄 알았습니다.  — 대규모 서버 관리자를 위한 로그인 관리 시스템을 다뤄보자는 것.

처음 이야기를 들었을 때 든 생각은 이거였습니다.

> "OAuth2 로그인 계정 관리 시스템? OAuth2는 뭐지?"
> 

좀 알아보니 OAuth2는 다른 서비스의 계정을 통해 로그인을 인증하는 프로토콜입니다. 하지만 OAuth2는 계정을 도둑맞았는데, 시스템 입장에서는 아무 일도 일어나지 않은 것처럼 인식하는 것이 문제라고 생각했습니다. 그래서 프로젝트 방향성을 함께 구체화하기 시작했습니다.

# 핵심 : 사람을 인증

저희 프로젝트 PickStranger의 한 줄 요약은 이렇습니다.

> "OAuth2 가 계정을 인증한다면, 이 시스템은 그 뒤에 있는 사람을 인증한다."
> 

기존의 방식을 대체하거나 교체하려고 하는게 아닙니다. 이미 잘 작동하는 로그인 흐름 위에, 하나의 안전장치를 얹는 방식을 택했습니다.

핵심 아이디어는 단순합니다. 

1. 로그인이 성공적으로 끝난 뒤에도 감시를 멈추지 않는다.
2. 브라우저/기기 핑거프린트(User-Agent, IP 등)로 “이 사람이 평소 쓰던 환경이 맞는지”를 확인
3. AI모델이 접속 패턴(시간, 위치, 기기)을 학습해서 평소와 다른 이상 징후를 잡아낸다.
4. 위험도가 높으면 세션을 즉시 차단하고 토큰을 만료시킨다.

즉, PickStranger는 “문을 열고 들어온 사람이 평소 그 집 사람이 맞는지”를 계속 지켜보는 역할입니다.

# 왜 하필 리버스 프록시인가

이 아이디어를 구현하는 방법은 여러 가지가 있었을 겁니다. 서비스 코드 안에 직접 로직을 심을 수도 있었습니다. 하지만 리버스 프록시 형태의 add-on 미들웨어 로 만들기로 했습니다. 이유는 크게 두 가지입니다.

첫째, 기존 서비스를 건드리지 않아야 실용적이다. 이미 운영 중인 서비스에 “이 보안 기능을 사용하려면 코드 싹 다 뜯어고쳐야 합니다.”라고 하면 아무도 안 씁니다. 리버스 프록시로 앞단에서 트래픽을 가로채는 방식이면, 서비스 코드를 거의 건드리지 않고도 보안 계층을 얹을 수 있습니다.

둘째, 관심사 분리가 명확해진다. 인증/인가는 여전히 OAuth2와 서비스 서버가 담당하고, “이 사람이 맞는지 의심스럽다.”는 판단은 프록시 계층에서 독립적으로 처리합니다. 한쪽에 장애가 나도 다른 쪽 서비스 전체가 멈추는 일이 없도록 설계하는 게 목표입니다.

# 팀 구성과 제 역할

PickStranger는 원래 아이디어를 낸 후배의 졸업 프로젝트를 제가 합류하여 도와주면서 팀프로젝트가 되었습니다. 역할은 명확히 나눠서 진행하고 있습니다.

- 리버스 프록시(Go) — 제가 맡은 파트. 트래픽을 가로채서 핑거프린팅하고, AI 분석 결과에 따라 세션을 차단하는 핵심 에이전트
- SDK (Kotlin/Java) — 팀원이 맡은 파트. 서비스 서버가 OAuth2 로그인 완료 시점을 프록시에 알려주는 연동 계층
- AI 이상 탐지 모듈 (Python) — 팀원이 맡은 파트, 접속 패턴을 학습해서 위험 점수를 산출하는 모델 서버

저는 이 중에서 Go 기반 리버스 프록시를 담당하고 있습니다. 평소 Go를 사용해본 적이 한 번도 없었는데, 이번 프로젝트를 goroutine 기반 동시성 모델과 대용량 트래픽 처리를 직접 경험해보는 것을 목표로 맡았습니다.
