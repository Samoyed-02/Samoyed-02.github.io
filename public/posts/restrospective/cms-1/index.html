배경#

BE 파트장이 부재인 상태로, 기능 명세서를 기반으로 한 Entity 설계 → ERD 작성 → API 명세서 작성까지 이어지는 일련의 작업을 진행하게 되었다. 다른 팀의 재촉을 받으면서 진행했는데, 막상 실무를 진행하다 보니 예상치 못한 곳에서 발목을 잡혔다.

진행 과정#

1. Entity 설계 & ERD 작성#

기획팀이 작성한 관리자/사용자 페이지 기능 명세서(v1)을 기준으로 엔티티를 뽑고 ERD를 작성했다. 여기까지는 명세서에 명시된 조회 정보, 등록 정보 항목들을 그대로 테이블 칼럼으로 매핑하는 작업이라 비교적 수월하게 진행했다.

2. API 명세서 작성 - 1차 중단#

ERD를 마치고 바로 API명세서 작성으로 넘어갔는데, 작성 하는 도중 정책사항과 기능 명세서 내용이 일치하지 않는 부분을 여러 군데에서 발견했다. 기획팀에 확인해보니 돌아온 답은 뜻밖에도 “정책사항이 아직 확정되지 않았다”는 것

여기서 문제를 하나 더 짚을 수 밖에 없었다. - 정책사항이 미확정이라는 건, 그 정책을 반영해서 작성된 기능 명세서 역시 확정된 문서가 아니라는 뜻이었다. 확정되지 않은 문서를 기준으로 API 명세서를 계속 작성하는 건 의밈가 없다고 판단해 작업을 일단 멈췄다.

3. API 명세서 작성 - 재개, 그러나 또 불일치#

다음 날 정책사항이 확정되었다는 연락을 받고 API 명세서 수정을 재개했다. 하지만 수정해보니 정책사항과 기능 명세서가 또다시 맞지 않는 부분이 들어났다. 게다가 세부적인 사항이 모호하거나 아예 명시되지 않은 부분도 다수 발견되면서, 하나하나 확인하고 판단해가며 작성해야 했다.

결과적으로 API 명세서는 아직 완료하지 못한 상태다.

어려었던 점#

  • 기존 문서 간의 정합성 부재: 정책 사항과 기능 명세서가 서로 다른 시점에, 다른 맥락으로 업데이트 되면서 두 문서가 계속 어긋났다. 어느 쪽을 기준으로 삼아야 할지 판단이 필요한 순간이 반복됐다.
  • 확정과 미확정의 경계가 불명확: “정책이 확정됐다”는 말을 듣고 작업을 재개했지만, 실제로는 완전히 확정된 상태가 아니었던 것으로 보인다. 확정 여부를 재확인하는 절차 없이 진행 신호만 받고 움직인 점이 비효율을 만들었다.
  • 명세서의 디테일 부족: 상위 기능/설명 수준에서는 명확했지만, API 스펙 수준(요청/응답 필드, 예외 처리, 상태값 등) 까지 내려가면서 명세서에 없는 정보가 많아 판단과 확인이 계속 필요했다.

느낀 점/ 배운 점#

  • 기능 명세서 작성이 완료되었다고 해서 그것이 곧 “확정”을 의 미하지는 않는다는 걸 체감했다. 정책 확정 → 기능 명세 확정 → 개발 문서 작성의 순서가 지켜지지 않으면, 뒷단 작업(ERD, API명세)에서 반복적으로 되돌아가는 비용이 발생한다.
  • “확정됐다”는 말을 그대로 신뢰하기 보다, 어느 항목이 확정됐는지 구체적으로 확인하는 습관이 필요하다는 걸 느꼇다.
  • 정책사항과 기능 명세서의 불일치를 발견했을 때 바로 작업을 멈추고 확인 절차를 거친 판단은 결과적으로 옳았다 — 확정 되지 않은 기준으로 계속 작업했다면 더 큰 리워크가 발생했을 것이다.
